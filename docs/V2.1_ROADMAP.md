# Stima V2 Roadmap Rewrite (Repo-Grounded Refinement)
_Date: 2026-04-19_

## Purpose

This roadmap narrows V2 to a focused plan centered on:

- capture first
- refine second
- lighter review and preview UX
- clearer line-item editing
- explicit reuse without product bloat

V2 is not about broad feature expansion. It is about making the current workflow calmer, faster, and easier to reuse while staying aligned with the actual app architecture.

---

## Product Position V2 Should Protect

Stima should continue to be:

> A lightweight, mobile-first quote drafting app for solo tradespeople and very small home-service operators that turns rough voice/text job notes into a professional draft, then lets the user quickly refine and deliver it.

V2 should compete on:

- simpler workflow
- lower clutter
- clearer editing
- faster repeat quoting
- polished delivery

V2 should not compete on:

- broad CRM/FSM feature breadth
- scheduling or dispatch
- heavy operations software
- complex estimating structures before they are needed

---

## Current Repo Shape To Respect

V2 should build on these existing implementation realities:

- Capture routes into persisted draft review through `/quotes/capture` and `/documents/:id/edit`.
- Review/edit is already the canonical editing surface for quote and invoice documents.
- The review screen uses a local session draft and only persists to the backend on Save Draft or Continue.
- Preview is split into header actions, preview actions, quote detail, line items, and linked invoice display components.
- Invoices are linked to source quotes through existing document relationships.
- Quote list search is currently frontend-side filtering over loaded quote summaries.
- The quote list endpoint intentionally omits line items.
- Backend quote behavior is already split into creation, mutation, deletion, outcome, share, and PDF artifact services.
- There is no catalog model or API yet.

Implications:

- Do not create a second quote-builder workflow.
- Do not turn line-item sheet dismiss into immediate server persistence.
- Do not overload the existing quote list API with chooser-only line-item preview data.
- Add duplication as a quote creation path, not as template infrastructure.
- Add catalog as a small user-owned resource, not as hidden document metadata.

---

## Product Rules Locked For V2

1. **Capture-first remains the product spine**
   - capture
   - review/edit
   - preview
   - share/send/convert

2. **Review remains the canonical editing surface**
   - no second quote-builder flow
   - no competing edit surfaces
   - line-item edits update the local review draft first

3. **Stima stays lighter than a full FSM/CRM**
   - no scheduling
   - no dispatch
   - no team-management expansion
   - no broad CRM growth

4. **New work must reduce repeat effort without cluttering the main workflow**
   - every new surface must earn its space
   - main review stays focused on the current draft

5. **Reuse should be visible and user-controlled**
   - no hidden draft mutation
   - no post-draft AI reshaping
   - reuse should be explicitly inserted or selected by the user

6. **Extra complexity belongs in secondary surfaces**
   - sheets
   - overlays
   - overflow menus
   - settings sub-screens
   - not permanent new sections on busy screens

---

## V2 Thesis

**V2 = cleaner editing first, explicit reuse second.**

That means:

- simplify the review UI
- simplify the preview UI
- improve line-item interaction
- add line-item catalog reuse
- add whole-quote reuse from existing quotes
- defer standalone quote templates in initial V2

# Locked UI/UX Decisions

## 1. Review header

- remove the top-right **X**
- keep the back arrow
- rely on existing back-target and leave-warning behavior

Repo note:

- The current workflow header supports a home/exit action. For review, V2 should suppress that action rather than changing route behavior.

## 2. Preview header

- keep **Edit** visible in the header
- keep secondary actions in overflow

Repo note:

- Preview already has a dedicated header action component, so this should stay a local preview action hierarchy change.

## 3. Invoice conversion on preview

- move **Convert to Invoice** into the overflow menu
- when no linked invoice exists, do not show a large empty linked-invoice card
- when a linked invoice exists, show a smaller linked-invoice surface only

Repo note:

- Conversion can keep using the existing conversion hook and recovery behavior; only the trigger and display hierarchy should change.

## 4. Linked invoice display when present

Show only:

- invoice number
- status
- due date
- total
- right-side arrow/open affordance

Remove:

- created date
- separate bottom "Open invoice" action

Treat the whole compact row/card as tappable.

Repo note:

- The API may continue returning `created_at` for compatibility. The V2 UI should simply stop displaying it in this surface.

## 5. Review line-item list

Default state:

- remove item count label
- show **Reorder** in the section header
- no drag handles
- no overflow menu
- no chevron
- tap row to edit

Reorder state:

- **Reorder** becomes **Done**
- drag handles appear
- row editing is disabled
- reorder changes apply immediately to the local review draft
- exiting reorder mode hides drag handles again

Repo note:

- Current reorder already mutates the local draft. V2 should add explicit reorder mode and default-row simplification without changing backend persistence timing.

## 6. Reorder interaction guidance

Initial V2 should not ship with a "snap with no feedback" feel.

Reorder mode should include a lightweight motion treatment:

- picked-up row should visibly look active/held
- surrounding rows should visibly make room during drag
- dropped row should settle clearly into place

Exact animation implementation is spec/task-level work, but the UX requirement is:

**reordering must feel intentional and legible, not instant and ambiguous.**

## 7. Line-item edit sheet

- no Save button
- no Cancel button
- no explicit close button
- standard sheet drag handle remains
- swipe down and tap outside behave the same
- dismissing valid edits autosaves into the local review draft
- invalid state blocks dismiss
- description required
- invalid price blocks dismiss with inline validation
- trash icon in top-right for delete while editing an existing item
- delete remains behind confirm
- details field reduced from 3 rows to 2 rows by default

Repo-grounded boundary:

- Autosave means "commit to the local review draft/session state."
- Autosave does not mean immediate backend persistence.
- Backend persistence remains Save Draft or Continue.

## 8. Customer assignment surface

Do not expand the assign customer modal with customer context in initial V2.

Keep it focused on:

- search
- select
- create inline

# Locked Reuse Decisions

## 9. Line-item reuse placement

Line-item reuse lives inside the existing add-line-item flow.

**Add Line Item** opens a tabbed sheet:

- `Manual`
- `Catalog`

Behavior:

- open on **Manual** by default
- load catalog when user switches tabs
- selected catalog item inserts as a normal editable line item into the local review draft

## 10. Whole-quote reuse placement

Whole-quote reuse lives in quote creation entry points.

Entry points:

- home FAB
- customer-level create-document entry

Entry sheet:

- **Create new**
- **Create from existing**

Repo note:

- "Create new" should continue into the existing capture/manual draft path.
- "Create from existing" should create a duplicated draft and route directly to `/documents/:id/edit`.

## 11. Create-from-existing chooser

Initial V2 whole-quote reuse comes from existing quotes, not standalone quote templates.

Chooser structure:

- `Recent`
- `All Quotes`

Search:

- reuse the same basic search behavior/pattern already used for quotes on home
- apply it to the chooser, not by reusing the exact home-row UI

Each quote card should show:

- title or doc number fallback
- customer name
- total
- date
- first 2-3 line items with description and price
- `+X more items` when applicable

Selecting a quote:

- creates a new draft
- keeps the same customer
- lands directly in review/edit
- does not route through capture

Repo-grounded adjustment:

- The existing quote list endpoint should not be expanded casually because it intentionally omits line items.
- Add a chooser-specific/reuse-candidate query shape for compact line-item context.

## 12. Duplicate quote semantics

Duplicated draft should keep:

- customer
- title
- line items
- pricing fields
- customer notes
- reusable quote content

Duplicated draft should not keep:

- status
- share state
- linked invoice/source invoice relationship
- PDF artifact state
- outcome state
- extraction review metadata

Open implementation decision:

- Whether to copy transcript is not yet locked. Recommended initial V2 default: do not copy extraction transcript unless product explicitly treats it as reusable quote content.

## 13. Quote templates

**Standalone quote templates are out of initial V2 scope.**

Initial V2 whole-quote reuse should use:

- existing real quotes
- line-item catalog reuse

Do not add:

- template models
- template naming flows
- template management
- template/gallery IA

# Catalog Decisions

## 14. Initial catalog save path

Initial V2 catalog creation should come from saving an existing line item.

Recommended affordance:

- bookmark/save-to-catalog action in the line-item edit sheet
- success toast after save

Repo-grounded adjustment:

- Saving to catalog is a backend write, but it must not force-save the current document draft.
- The document draft remains local until Save Draft or Continue.

## 15. Catalog management location

Keep bottom nav unchanged.

Catalog management should live under:

- **Settings -> Line Item Catalog**

Do not add a new bottom-nav destination in initial V2.

Repo-grounded adjustment:

- Prefer a focused settings sub-screen/component over adding more responsibilities to the already large settings profile form.

## 16. Initial catalog fields

Initial catalog item fields:

- title
- details
- default price

Recommended model:

- user-owned catalog item
- `title` required
- `details` optional
- `default_price` nullable
- created/updated timestamps

Defer:

- usage count
- recent use ranking
- categories
- tags
- richer structure
- quantity/unit/rate fields

# Recommended V2 Build Order

## Phase 1 - UX simplification pass

1. Remove review-screen X.
2. Keep preview Edit visible and simplify preview action hierarchy.
3. Move Convert to Invoice into overflow.
4. Hide linked invoice UI when no invoice exists.
5. Simplify linked invoice display when invoice exists.
6. Simplify line-item rows in default mode.
7. Add explicit Reorder / Done mode.
8. Improve reorder motion feedback.
9. Update line-item edit sheet to local-draft autosave on valid dismiss.

Reason:

- These changes fit the current frontend architecture and reduce clutter before adding reuse UI.

## Phase 2 - backend contracts for whole-quote reuse

10. Add duplicate quote draft endpoint/service behavior.
11. Add quote reuse-candidate query shape for chooser cards.

Reason:

- Frontend chooser work needs stable backend contracts to avoid N+1 detail fetching and later rework.

## Phase 3 - whole-quote reuse frontend

12. Add `Create new | Create from existing` entry sheet.
13. Add `Recent | All Quotes` chooser with compact quote context.
14. Duplicate selected quote into a new draft and land in review.

Reason:

- This keeps duplication in creation paths and avoids turning preview/review into template management.

## Phase 4 - line-item catalog backend

15. Add line-item catalog model, migration, schemas, repository, service, and API.

Reason:

- Catalog requires new persistence and should land before UI integration.

## Phase 5 - line-item catalog frontend

16. Add bookmark/save-to-catalog flow from line-item edit sheet.
17. Add `Manual | Catalog` Add Line Item sheet.
18. Support selecting catalog items into the local draft.
19. Add Settings -> Line Item Catalog management surface.

Reason:

- This contains catalog reuse in secondary surfaces and avoids global navigation churn.

## Phase 6 - later evaluation

20. Evaluate internal customer notes on customer detail page.
21. Evaluate filters/search improvements.
22. Reassess whether true quote templates are actually needed.

- CRM pipeline features
- heavy estimate-builder structures
- proposal sections / alternates / add-ons
- hidden AI draft mutation after extraction
- quantity/unit/rate systems
- catalog categories/tags

---

# Success Criteria

## Workflow simplicity

- review screen feels visibly calmer
- preview screen contains fewer secondary blocks
- line-item editing feels faster and cleaner
- reorder interaction feels legible rather than abrupt

## Reuse value

- users can insert common line items quickly without cluttering review
- users can recreate a quote from prior work with enough context to choose confidently

## Product clarity

- Stima still feels like one product with one workflow
- reuse does not create a second mental model or heavier product surface
- catalog and duplication do not drift into template infrastructure

---

# Final Summary

This roadmap narrows V2 to:

- protect the lightweight product shape
- simplify the densest screens
- make line-item editing cleaner
- add explicit reuse where it naturally fits
- defer standalone quote templates until there is evidence they are needed

That keeps Stima aligned with its strongest differentiation:

**fast capture, calm review, polished delivery, and lower software heaviness than larger alternatives.**
